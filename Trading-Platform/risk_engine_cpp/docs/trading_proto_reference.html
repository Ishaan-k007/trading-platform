trading.proto — Complete Reference

Trading Platform — gRPC Contract Between Flask/Python and the C++ Engine  |  Date: 2026-07-15

Part 1: What This File Is, In Plain English

trading.proto is not code that runs. It is a shared vocabulary — a description of the exact shape data must have whenever Python (Flask, the Binance feed handler) and C++ (risk_engine.exe) exchange information. Python and C++ are two completely separate programs that never share memory; the only way they can talk to each other is by agreeing in advance, in writing, on what a message looks like. This file is that agreement.

Both sides run this same file through the protobuf compiler (protoc) and get real, usable code generated from it — actual classes in Python (services/trading_pb2.py) and actual classes in C++ (generated into the CMake build directory). Because both sides generate their code from the identical source file, it is structurally impossible for them to disagree about what a message contains.

The building blocks used throughout this file

ConceptWhat it means
messageDeclares a data structure — like defining a small class with only these fields and nothing else.
= 1, = 2, ...Not a default value or array index. A permanent field tag — an ID number protobuf uses on the wire instead of sending the field name as text every time. Scoped per-message: message A's field 1 and message B's field 1 are unrelated. This is part of why protobuf is smaller and faster than JSON.
repeatedMeans "a list of," not a single value. A repeated PriceLevel field becomes a list-like object on both sides that you can iterate or append to.
service / rpcDeclares a remote procedure — a function one process can call on another process over the network. Each rpc line names exactly one request message type and one response message type.
Request / Response pairThe standard shape for every RPC in this file: one message travels out to the callee, a different message travels back. Same pattern as a function's arguments and its return value, just over a network instead of in-process.
Every RPC in this file is unary — one request, one response, no streaming. This matches the actual traffic pattern needed: even the highest-volume RPC (UpdateOrderBook, at full symbol-universe scale) is on the order of a few thousand calls/second, which a plain unary call over an already-open gRPC connection handles comfortably. Streaming RPCs exist in gRPC but were deliberately not used here — they solve a different problem (very high-frequency continuous push, or client-finishes-then-gets-one-summary) that doesn't apply to any current use case.

Part 2: Shared Building Blocks

PositionEntry

Used inside LoadUserRequest

Represents one holding a user has in one symbol — not sent on its own, only ever as part of a list inside a bigger message.

FieldTypeMeaning
symbolstringTicker, e.g. "AAPL"
quantitydoubleHow many shares/units are held
average_pricedoubleAverage cost basis for this position
Example instance: { symbol: "AAPL", quantity: 10, average_price: 182.50 }

PriceLevel

Used inside OrderBookUpdateRequest

Represents one row of an order book: at this price, this much quantity is resting. Like PositionEntry, it is a building block — never sent alone, always as part of a repeated list.

FieldTypeMeaning
pricedoubleThe price at this level
quantitydoubleTotal quantity resting at that price
Example instance: { price: 67200.50, quantity: 0.42 }
  — meaning: someone has an order for 0.42 BTC at $67,200.50

Part 3: Every RPC, In Detail

1. CheckOrder

Called by: Flask (order_service.py)  →  Answered by: C++ risk engine

What it's for: Before any trade is allowed to happen, Flask asks the C++ engine "can this user actually do this?" — checking funds for a BUY or held quantity for a SELL — entirely from in-memory state, no database round trip. This is the risk-check step of the order pipeline.

CheckOrderRequest

FieldTypeMeaning
user_idint32Who is placing the order
order_idstringClient-supplied unique ID (idempotency)
symbolstringWhat's being traded
sidestring"BUY" or "SELL"
order_typestring"MARKET" or "LIMIT"
quantitydoubleHow much to trade
limit_pricedoubleOnly meaningful for LIMIT orders

CheckOrderResponse

FieldTypeMeaning
approvedboolCan this order proceed?
reasonstringWhy it was rejected, if it was
risk_scoredoubleInternal risk metric
fill_pricedoubleThe price the order would fill at if approved

Connects to: If approved is true, Flask immediately calls UpdateState to actually apply the trade. If false, order_service.py raises InsufficientFundsError or InsufficientPositionError using the returned reason/fill_price.

2. UpdateState

Called by: Flask, immediately after an approved CheckOrder

What it's for: Commits an approved trade into the engine's in-memory state (new cash balance, new position) and writes a Write-Ahead-Log entry — this is the only place a trade becomes "real" inside the engine. No PostgreSQL write happens here or anywhere in this path; persistence happens later, asynchronously, via the WAL → Kafka → fill_consumer.py pipeline.

UpdateStateRequest

FieldTypeMeaning
user_idint32Whose state to update
symbolstringWhich symbol was traded
new_cash_balancedoubleCash balance after this fill
new_quantitydoublePosition size after this fill
new_average_pricedoubleUpdated cost basis
order_id, side, fill_price, quantity, order_typemixedFull fill details — included so the WAL entry written from this call is complete on its own, without needing to look anything else up

UpdateStateResponse

FieldTypeMeaning
successboolDid the update apply
messagestringHuman-readable detail

Connects to: the WAL entry written here is later read by pipeline/wal_producer.py and published to Kafka's order-fills topic, eventually landing in PostgreSQL via pipeline/fill_consumer.py.

3. LoadUser

Called by: Flask, only the first time a user is seen

What it's for: The C++ engine holds trading state in RAM, not the database — but RAM is empty on startup. The first time a given user places an order, Flask reads their current cash and positions out of PostgreSQL and hands them to the engine once, so every subsequent order for that user is answered from memory with no DB read.

LoadUserRequest

FieldTypeMeaning
user_idint32Who to load
cash_balancedoubleStarting cash, from the accounts table
positionsrepeated PositionEntryAll current holdings, from the positions table

LoadUserResponse

FieldTypeMeaning
successboolDid the load apply
messagestringHuman-readable detail

Connects to: gated by HasUser (below) — Flask always checks HasUser first and only calls LoadUser if the answer is no. This pair together is the "write-through cache" pattern.

4. HasUser

Called by: Flask, before every order, to decide whether LoadUser is needed

What it's for: A cheap yes/no check — "is this user already sitting in your memory?" Exists purely to avoid an unnecessary PostgreSQL read on every single order; only the very first order for a given user pays that cost.

HasUserRequest

FieldTypeMeaning
user_idint32Who to check

HasUserResponse

FieldTypeMeaning
loadedboolTrue if already in memory

5. GetPrice

Called by: Flask market_routes.py (GET /prices/<symbol>), and internally by order_service.py

What it's for: Returns the current price the engine is using for one symbol. Today this is a GBM-simulated price; from Stage 3 onward it will read from the real order book instead — this RPC's shape doesn't need to change for that, only what's behind it inside the engine.

GetPriceRequest

FieldTypeMeaning
symbolstringWhich symbol

GetPriceResponse

FieldTypeMeaning
symbolstringEchoed back
pricedoubleCurrent price
updated_atstringTimestamp of the last update
If the symbol was never seeded into the engine, this RPC returns a gRPC NOT_FOUND status, which risk_engine_client.py turns into a SymbolNotFoundError — this is the exact mechanism that currently blocks trading on any symbol the engine hasn't been told about in advance. Relevant once real Binance symbols (hundreds of them) start arriving — see the UpdateOrderBook section below.

6. GetAllPrices

Called by: Flask market_routes.py (GET /prices)

What it's for: Same idea as GetPrice, but for every symbol at once — powers the "all current prices" market data endpoint.

GetAllPricesRequest

Empty — no input needed, it's "give me everything you have."

GetAllPricesResponse

FieldTypeMeaning
pricesmap<string, double>Symbol → current price, for every symbol the engine knows about

7. UpdateOrderBook (new — Stage 1)

Called by: the Binance feed handler (pipeline/binance_feed_producer.py), once per depth snapshot

What it's for: Delivers one real order book snapshot for one symbol — Binance's top-20 bid/ask levels — from the Python feed handler into the C++ engine's memory, replacing the fake GBM-simulated price with real market data.

OrderBookUpdateRequest

FieldTypeMeaning
symbolstringWhich trading pair this snapshot is for, e.g. "BTCUSDT"
bidsrepeated PriceLevelUp to 20 buy-side levels, best-to-worst
asksrepeated PriceLevelUp to 20 sell-side levels, best-to-worst
updated_atstringTimestamp of this snapshot

OrderBookUpdateResponse

FieldTypeMeaning
successboolDid the update apply
Example instance sent for one BTCUSDT tick:

OrderBookUpdateRequest {
  symbol: "BTCUSDT"
  bids: [ {price: 67200.50, quantity: 0.42}, {price: 67199.00, quantity: 1.10}, ... up to 20 ]
  asks: [ {price: 67201.00, quantity: 0.30}, {price: 67202.50, quantity: 0.85}, ... up to 20 ]
  updated_at: "2026-07-15T14:02:31Z"
}
Status: contract defined, handler not yet written. The .proto definition above exists and Python's generated stubs are ready to use it, but there is no C++ function implementing UpdateOrderBook yet, and no OrderBook store to hold the data it delivers (Stage 2). Because the stream used is Binance's partial-depth snapshot (not diff-depth), each incoming message is a complete, self-contained picture — the eventual handler only needs to replace whatever it's holding for that symbol, never merge or reconcile against previous state.
Auto-registration requirement: unlike PriceStore (which requires a symbol to be pre-seeded from the market_prices table before GetPrice/set_price will do anything for it), the new OrderBook store must create an entry for a symbol the first time UpdateOrderBook is called for it. At target scale (300+ live Binance symbols), nobody is hand-inserting hundreds of rows into Postgres to unlock trading — the real order book data itself is the registration.

Part 4: How The RPCs Fit Together — End to End

Order placement flow

Flask receives POST /orders
   → HasUser?           (cheap check: is this user already in memory?)
       → no  → LoadUser  (one-time: pull cash + positions from Postgres)
   → CheckOrder          (can they afford / do they hold enough?)
       → approved? → UpdateState   (commit the fill, write WAL entry)
                   → (async, off this path) WAL → Kafka → Postgres

Market data flow

Browser / researcher hits GET /prices or GET /prices/<symbol>
   → Flask calls GetAllPrices or GetPrice
   → today: reads the GBM-simulated PriceStore
   → after Stage 3: reads the real OrderBook store instead — same RPCs, different data source underneath

Real market data ingestion flow (Stage 1, in progress)

Binance WebSocket (partial-depth stream, sharded across multiple connections)
   → Python feed handler parses the JSON snapshot
   → builds an OrderBookUpdateRequest
   → calls UpdateOrderBook over gRPC
   → [not yet built] C++ handler stores it in the OrderBook class (Stage 2)
   → [not yet built] GetPrice / a future signals RPC reads real data from there
