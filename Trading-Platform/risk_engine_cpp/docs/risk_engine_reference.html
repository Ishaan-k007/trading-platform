Part 1: Architecture Decisions

Why C++ and not Python for the risk engine

Misunderstanding: "Can't Python hold data in RAM too?"

Yes — Python can hold a dict of user cash in memory. The problem is not the language, it is Flask's deployment model.

Flask runs as multiple separate processes, not threads. Each process has its own copy of memory that is never shared with the others:

gunicorn --workers 4 app:app

Process A  →  user_cash = {1: £1000}   ← separate copy in RAM
Process B  →  user_cash = {1: £1000}   ← separate copy in RAM
Process C  →  user_cash = {1: £1000}   ← separate copy in RAM

Process A deducts £500. Processes B and C still think the user has £1000. They share no memory. This is a correctness bug — not a performance issue.

The C++ gRPC server is one single process. All Flask workers connect to port 50051 over localhost. There is only ever one copy of user state in memory.

Python also has the GIL (Global Interpreter Lock). Even with threads (not processes), Python can only execute one thread at a time. C++ threads run truly in parallel on separate CPU cores with no such restriction.

The real speed gain is not C++ arithmetic. A risk check in Python that queries PostgreSQL takes 5–20ms. A C++ memory check takes 0.001ms. Skipping the database is the win — not faster multiplication.

Why not Kafka for risk checks

Kafka is async — you publish a message and move on without waiting. CheckOrder is synchronous — Flask blocks waiting for approve or reject before proceeding with the fill. You cannot use a fire-and-forget system to answer a question you are waiting on.

Kafka also adds 5–15ms minimum round-trip latency from the broker. On the hot path of every trade this is unacceptable when the whole point is sub-millisecond risk checks.

Where Kafka is genuinely useful in trading: broadcasting fills to multiple consumers — audit log, P&L calculator, notification service, analytics. One event, many recipients, no synchronous response needed. This is a Stage 5–6 addition to this platform.

The Race Condition and How We Solve It

The problem

User has £1000

Thread A: reads £1000  →  approves £600 buy  →  returns
Thread B: reads £1000  →  approves £600 buy  →  returns
          ↑ both read before either updates memory
          ↑ user ends up with -£200

UpdateState only runs after Flask commits to the database — too late to prevent the second approval from seeing the stale value.

The fix: preemptive subtraction

Inside check_and_reserve_position, while holding the per-user lock, we subtract the funds before returning the approval:

Thread A: LOCK user 1 → reads £1000 → subtracts £600 → cash now £400 → UNLOCK → approved
Thread B: LOCK user 1 → reads £400  → £400 < £600   → UNLOCK → rejected

Thread B sees the already-deducted value. The race condition is eliminated in C++ memory with no database involvement.

When update_position runs

Flask commits the fill to PostgreSQL, then calls update_position with the authoritative database values. C++ overwrites its tentative subtraction with the real number. If the fill failed, Flask sends the original balance and C++ restores it.

Why send full values not deltas

Deltas can drift:
C++ has £400 (tentative). Flask sends cash_delta = -600. C++ computes 400 + (-600) = -200. Wrong if the tentative was slightly off.
Full values cannot drift:
C++ has £400 (tentative). Flask sends new_cash = 400.00 (straight from DB). C++ sets 400.00. Always matches PostgreSQL exactly.

trading.proto

Contract File

The contract between Flask (Python) and the risk engine (C++). Neither language yet — a description in proto syntax that the protoc compiler reads to generate real code in both languages automatically.

Proto vs gRPC — they are different things

Protocol Buffers (proto) — defines the messages and the service contract. Handles serialisation of data into bytes.

gRPC — the transport layer. Handles network communication — TCP connections, HTTP/2, routing requests to your service methods.

Proto defines what to send. gRPC handles how to send it. You cannot use one without the other.

Field numbers

message CheckOrderRequest {
    string symbol = 1;   // ← not an array index
    double price   = 2;  // ← a permanent binary identifier
}

Proto serialises to binary using these numbers, not field names. Once assigned, never change them — even if you rename the field. Numbers 1–15 use 1 byte; 16–2047 use 2 bytes. Put your most-used fields first.

Key design decisions in the proto file

fill_price in CheckOrderResponse — C++ returns the exact price it used for the risk check. Flask uses this price for the fill. Eliminates the stale price problem where check and fill use different prices.

Authoritative values in UpdateStateRequest — new_cash_balance, new_quantity, new_average_price come straight from PostgreSQL after the fill commits. C++ overwrites, never accumulates deltas.

repeated PositionEntry in LoadUserRequest — repeated means a list. All of a user's positions sent in one call at login time.

The service block

service TradingService {
    rpc CheckOrder   (CheckOrderRequest)   returns (CheckOrderResponse);
    rpc UpdateState  (UpdateStateRequest)  returns (UpdateStateResponse);
    rpc LoadUser     (LoadUserRequest)     returns (LoadUserResponse);
    rpc GetPrice     (GetPriceRequest)     returns (GetPriceResponse);
    rpc GetAllPrices (GetAllPricesRequest) returns (GetAllPricesResponse);
}

This tells protoc "these five callable functions exist." Protoc generates an abstract C++ base class with five virtual functions. You never write the base class — you write the implementation in trading_service.cpp by inheriting from it.

gRPC function signatures — why Status and not the response type

Status CheckOrder(ServerContext* ctx,
                  const CheckOrderRequest* request,
                  CheckOrderResponse* response);

The CheckOrderResponse IS returned — but via the output pointer parameter, not the return value. You fill in the response object inside the function. The Status return value is at the framework level — it tells gRPC whether the function executed successfully (OK) or failed at a network/system level (INTERNAL_ERROR, NOT_FOUND etc.). Your business logic result (approved/rejected) goes inside the response object.

price_store.hpp + price_store.cpp

In-Memory Price Cache

Thread-safe dictionary mapping symbol strings to PriceData (price, volatility, drift, updated_at). Lives entirely in RAM. The GBM simulation thread writes to it every second. gRPC server threads read from it on every risk check.

Why shared_mutex not regular mutex

A regular mutex allows only one thread at a time — readers queue behind each other for no reason even though simultaneous reads are perfectly safe.

Regular mutex:
gRPC thread 1: LOCK → reads AAPL → UNLOCK
gRPC thread 2: WAITING...
gRPC thread 3: WAITING...       ← pointless, reads don't conflict

shared_mutex:
gRPC thread 1: shared_lock ✓   → reads AAPL
gRPC thread 2: shared_lock ✓   → reads MSFT  ← simultaneous, safe
GBM thread:    exclusive lock ✗ → waits for readers to finish
GBM thread:    exclusive lock ✓ → writes, all readers wait

Why mutable on the mutex

get and get_all are const methods — they promise not to modify any member variables. But locking a mutex modifies it internally. mutable is the exception clause: "this one field is allowed to be modified even inside a const method." Without it the compiler rejects locking the mutex inside a const function.

const at the end of method signatures

std::optional<PriceData> get(const std::string& symbol) const;
//                           ^                             ^
//                     parameter won't be modified    method won't modify the object

Why two methods — add_symbol vs set_price

add_symbol is called once per symbol at startup. It sets all four fields: price, volatility, drift, and updated_at. Volatility and drift are GBM parameters — constants loaded from PostgreSQL that never change during the simulation.

set_price is called every second by the GBM thread. It only updates price and timestamp. Passing volatility and drift 86,400 times per day for no reason would be wasteful and misleading about what is actually changing.

Iterator pattern in C++

auto it = prices.find("AAPL");
if (it != prices.end()) {      // end() is the "not found" sentinel
    it->second.price = 183.0;  // second = value (PriceData)
}                               // first  = key (symbol string)

Reference vs copy — the most common mistake

Wrong — modifies a copy, map unchanged:
PriceData data = it->second;
data.price = 183.0; // copy goes out of scope, map still has old price
Correct — reference to the real entry:
PriceData& data = it->second;
data.price = 183.0; // modifies the actual map entry

std::optional

If a symbol does not exist in the map you cannot return a PriceData — there is nothing to return. std::optional<PriceData> is a wrapper that either contains a value or contains nothing. Return std::nullopt for "not found." The caller must check for empty before using the value — the compiler enforces this.

user_state_store.hpp + user_state_store.cpp

In-Memory User State

Thread-safe store mapping user_id (int) to UserState (cash, per-user mutex, position map). Implements the full preemptive reservation logic for risk checks.

Why unique_ptr<mutex> not mutex directly

std::mutex cannot be copied or moved. When you insert UserState into an unordered_map, C++ tries to move it. Since UserState contains a mutex, the move fails to compile.

The heap vs stack:

The stack is automatic memory inside functions — destroyed when the function ends. The heap is permanent memory allocated explicitly — stays alive until freed. Objects on the heap have a fixed memory address that never changes.

How unique_ptr solves it:

unique_ptr<mutex> stores the mutex on the heap at a fixed address. The pointer itself is just a number (the address) which can be moved freely. Moving the pointer copies the number to a new location — the mutex stays at the same heap address and never knows it happened.

Stack:                    Heap:
┌──────────────┐         ┌─────────────┐
│ unique_ptr   │────────▶│   mutex     │  address: 0x7f3a2100 (fixed)
│ = 0x7f3a2100 │         │             │
└──────────────┘         └─────────────┘

After move: pointer moves, mutex stays at same address.

Why "unique": Only one unique_ptr can own a given object at a time. Ownership can be transferred (moved) but never shared. When the unique_ptr is destroyed, it automatically frees the object — no manual delete needed.

Dereferencing with *: The unique_ptr stores an address, not the mutex itself. To get the actual mutex you dereference with *:

std::unique_lock ul(*user.lock);  // *user.lock = the actual mutex

std::move

user_positions[user_id] = std::move(state);

Transfers ownership of state's contents into the map. After this line, state is an empty shell. The unique_ptr inside it now belongs to the map entry. Without std::move, C++ would try to copy state — copying a unique_ptr is forbidden (only one owner at a time).

Two-level locking in check_and_reserve_position

Step 1: shared_lock on user_positions_mutex   → safely find user in the map
Step 2: unique_lock on user.lock              → own this user's state exclusively
Step 3: unlock user_positions_mutex           → release map lock, keep per-user lock
Step 4: check cash / position and subtract    → atomic, no other thread can interfere

Step 3 is the efficiency insight: you only need the map lock long enough to find the user. Once you have the per-user lock, release the map lock so other threads can load different users simultaneously. Hold locks for the minimum time needed.

Part 4: Common Mistakes Reference

1. Redefining the class in the .cpp file

Wrong: Writing class PriceStore { ... } inside price_store.cpp
Correct: The class definition lives only in the .hpp. The .cpp only contains method implementations prefixed with ClassName::

2. Wrong or missing class prefix on methods

void load_user(...) { }
This is a free function — not a class method. The compiler cannot find member variables.
void UserStateStore::load_user(...) { }

3. Confusing UserState with UserStateStore

UserState — the struct holding one user's cash, mutex, and positions.
UserStateStore — the class managing all users. This is the one with the methods.

4. Modifying a copy instead of a reference

PriceData data = it->second; then data.price = x; — modifies a copy, map unchanged.
PriceData& data = it->second; then data.price = x; — modifies the real map entry.

5. Inverted find condition — the most repeated mistake

// Reading it: "if find result == the not-found sentinel, user doesn't exist"
if (map.find(id) == map.end())  return false;  // ← correct: not found → false
if (map.find(id) != map.end())  return false;  // ← wrong: found → false (backwards)

6. Variable named same as its type

std::shared_mutex shared_mutex; — type and variable share the same name, confuses IDE and compiler.
std::shared_mutex user_positions_mutex; — descriptive name, no conflict.

7. Trailing comma in parameter list

bool check(int id, double price, ) — syntax error.
bool check(int id, double price)

8. Semicolon after preprocessor directives

#pragma once;
#pragma once — preprocessor directives never end with semicolons.

9. Typo in include angle brackets

#include <shared_mutex>> — extra closing bracket.
#include <shared_mutex>

10. Missing semicolon after unique_lock declaration

std::unique_lock lock2{*user.lock}
std::unique_lock lock2{*user.lock};

11. Missing return true after successful reservation

After subtracting cash or position quantity inside check_and_reserve_position, always return true. A bool function must always return a value on every path. Add return false at the very end as a fallback for unrecognised side values.

12. Cannot #include a .cpp file

Including a .cpp file copy-pastes its entire content into every file that includes it. If two files include the same .cpp, the compiler sees every class and function defined twice and throws duplicate definition errors.

The .hpp is the menu — everyone reads it. The .cpp is the kitchen — compiled once, hidden from everyone else. You never include the kitchen.
